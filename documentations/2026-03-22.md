# Connected — Build Log · 2026-03-22

## Project Overview

**Connected** is a university-exclusive web app built for Constructor University students. It provides a social and academic hub: courses, clubs, marketplace, events, and a campus news feed — all gated to verified `@constructor.university` email addresses.

---

## Stack

| Layer | Choice |
|---|---|
| Framework | Next.js App Router (TypeScript) |
| Styling | Tailwind CSS + shadcn/ui |
| Backend | Supabase (Auth + Postgres + Realtime) |
| Auth strategy | Magic link (default) + email/password |
| Hosting target | Vercel |

---

## What Has Been Built

### 1. Project Foundation

- Clean Next.js App Router structure with route groups: `(auth)`, `(main)`, `(onboarding)`
- Each route group has its own `layout.tsx` — no URL impact, just isolated chrome
- `tailwind.config.ts` and `globals.css` with full Connected design system:
  - Primary: Deep Indigo `#23389c`
  - Surface: `#f9f9f9`, background `#ffffff`
  - Fonts: Plus Jakarta Sans (`--font-display`) + Inter (`--font-body`)
  - Utilities: `.gradient-primary`, `.glass`, `.pb-safe`
- `app/layout.tsx` loads both fonts via `next/font/google`, sets mobile viewport + theme color

### 2. Supabase Setup

- `lib/supabase/client.ts` — browser client via `createBrowserClient` (no Database generic to avoid `never` inference issues)
- `lib/supabase/server.ts` — async server client via `createServerClient` with full cookie read/write handling
- `.env.local.example` with `NEXT_PUBLIC_SUPABASE_URL` and `NEXT_PUBLIC_SUPABASE_ANON_KEY`

### 3. Database Schema (`supabase/schema.sql`)

Full schema covering:

| Table | Purpose |
|---|---|
| `universities` | Approved domains; `constructor.university` is the seed |
| `profiles` | One row per user; linked to `auth.users` via trigger |
| `courses` | Course catalogue with ratings |
| `course_reviews` | Student reviews with upvote count |
| `course_questions` | Q&A per course |
| `course_answers` | Answers to questions |
| `clubs` | Campus clubs with member counts |
| `club_memberships` | Join table; trigger maintains `clubs.member_count` |
| `club_posts` | Feed posts inside clubs |
| `club_comments` | Comments on club posts |
| `marketplace_listings` | Student buy/sell listings |
| `marketplace_messages` | DM thread per listing |
| `events` | Campus events with RSVP count |
| `event_rsvps` | Join table; trigger maintains `events.rsvp_count` |
| `news_articles` | Campus news feed |

- `year_of_study` Postgres enum: `'1' | '2' | '3' | '4' | 'masters' | 'phd'`
- Row-Level Security enabled on every table, all policies scoped by `university_id`
- GIN indexes on `courses` and `clubs` for full-text / fuzzy search
- Supabase Realtime enabled on `marketplace_messages`, `club_posts`, `club_comments`

### 4. Seed (`supabase/seed.sql`)

```sql
insert into universities (name, domain)
values ('Constructor University', 'constructor.university')
on conflict (domain) do nothing;
```

Typed interfaces for every table:

### 6. Middleware (`middleware.ts`)

Six-case routing logic — zero DB queries per request:

| Case | Condition | Action |
|---|---|---|
| 1 | No session + protected route | Redirect `/login` |
| 2 | No session + auth route | Pass through |
| 3 | Session + auth route | Redirect `/home` |
| 4 | Session + `/onboarding` + already complete | Redirect `/home` |
| 5 | Session + protected + onboarding incomplete | Redirect `/onboarding` |
| 6 | Session + onboarding complete + any route | Pass through |

Onboarding state read from `user.user_metadata.onboarding_complete` (JWT) — no extra DB round-trip.

### 7. Authentication (`app/(auth)/`)

**Login page** (`/login`):
- Mode toggle: Magic link (default) ↔ Password
- Suffix-anchored email input — user types handle, `@constructor.university` is fixed
- Magic link: `signInWithOtp` with `shouldCreateUser: false`
- Password: `signInWithPassword`, then `window.location.href = "/home"` (middleware handles onboarding redirect)
- "Check your inbox" confirmation state after magic link send
- Client-side domain validation before hitting Supabase

**Signup page** (`/signup`):
- Same suffix-anchored input pattern
- University badge prominently displayed
- Server-side domain check via `universities` table (guards against client bypass)
- Passes `university_id` in user metadata for callback use
- Email confirmation screen after successful signup

**Auth callback** (`/auth/callback/route.ts`):
- Exchanges code for session
- Inserts stub `profiles` row if none exists (sets `university_id` from user metadata)
- Reads `onboarding_complete` from profile or user metadata
- Redirects to `/onboarding` or `/home`

### 8. Onboarding (`app/(onboarding)/`)

**Layout**: Minimal centered layout — no sidebar, no bottom nav.

**2-step form** (`/onboarding`):

- **Step 1**: Full name input
- **Step 2**: Faculty selection (7 Constructor University faculties), programme/major text input, year of study grid buttons
- Progress dots animate between steps
- `useTransition` wraps server action call for pending state

**Server action** (`actions.ts`):
1. Updates `profiles` row: `full_name`, `faculty`, `major`, `year_of_study`, `onboarding_complete: true`
2. Calls `supabase.auth.updateUser({ data: { onboarding_complete: true } })` — mirrors flag into JWT so middleware reads it without a DB query
3. Redirects to `/home`

## Key Design Decisions

**No Database generic on Supabase clients** — using `createBrowserClient()` and `createServerClient()` without `<Database>` generic. When the inferred types were wrong, explicit `as unknown as Type` casts were used in pages. This avoids cascading `never` type inference when the hand-written `Database` interface doesn't perfectly satisfy Supabase's internal constraints.

**onboarding_complete in JWT** — the flag lives in both `profiles.onboarding_complete` (source of truth) and `user.user_metadata.onboarding_complete` (JWT cache). Middleware reads the JWT copy to avoid a Supabase DB query on every navigation.

**Route groups for layout isolation** — `(auth)` uses a centered card layout, `(main)` uses sidebar + bottom nav, `(onboarding)` uses a minimal centered layout. None of these group names appear in URLs.

**Domain locking at two layers** — client-side suffix input prevents typos; server-side `universities` table lookup prevents bypassing the client check entirely.
